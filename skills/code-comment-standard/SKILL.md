# 代码标准化注释生成 Skill

**Skill ID**: CODE-COMMENT-STANDARD-V1.0
**版本**: V1.0.0
**更新时间**: 2026-04-07
**适用范围**: 全品类编程语言、全场景代码注释生成/补全/规范化整改
**核心受众**: Agent执行体、代码开发者、团队协作项目、开源项目维护者
**标签**: 代码规范、注释生成、可读性优化、可维护性提升

## 一、技能概述

本Skill用于标准化约束Agent的代码注释生成行为，明确不同层级、不同场景、不同编程语言的注释规范、执行流程、质量标准与禁止事项。核心目标是让Agent生成**精准无冗余、规范无歧义、匹配代码逻辑、符合行业标准**的代码注释，避免无效注释、错误注释、过度注释，提升代码的可读性、可维护性与团队协作效率。

## 二、适用场景

1. 全新开发代码的全量标准化注释添加
2. 存量代码的注释补全、格式规范化与内容优化
3. 团队协作/开源项目的代码注释标准化整改
4. 教学示例/培训代码的精细化注释拆解
5. 底层库/框架/SDK对外接口的文档级注释生成
6. 遗留代码的注释梳理与逻辑说明补全

## 三、核心执行目标

1. **精准匹配**：注释100%贴合代码逻辑，无错误、无误导、无歧义，代码修改时必须同步更新对应注释
2. **价值优先**：注释核心解释「为什么这么写」，而非重复「代码做了什么」，每一条注释都具备不可替代的信息价值
3. **规范统一**：严格遵循对应编程语言的官方注释规范，同一份代码内注释风格、格式、层级完全统一
4. **分层清晰**：按「文件级-模块级-类/结构体级-函数/方法级-逻辑块级-行内级」分层注释，层级边界清晰
5. **兼容可读**：兼顾新手开发者的理解成本与资深开发者的阅读效率，公共接口注释完整，内部实现注释精准

## 四、前置执行原则

Agent执行本Skill时，必须严格遵守以下优先级最高的核心原则：

1. **先理解，后注释**：必须先完整通读代码，拆解代码的整体功能、模块划分、核心逻辑、输入输出、依赖关系、异常处理、边界条件，完全理解代码语义后，再生成注释，禁止断章取义式注释
2. **风格一致性优先**：若目标代码已有统一的注释规范/风格，优先遵循原有风格，仅补全缺失内容、修正错误注释，禁止强行替换原有注释风格
3. **不修改代码逻辑**：注释生成过程中，仅可添加/修改/删除注释内容，绝对禁止修改原有代码的逻辑、格式、缩进、变量名与结构
4. **面向维护者视角**：所有注释均面向未来的代码维护者，而非当下的编写者，重点补充代码中无法直接体现的设计意图、踩坑记录、边界约束、兼容处理
5. **最小必要原则**：可写可不写的注释不写，能通过代码语义直接理解的内容不写，仅对非显而易见的核心逻辑、特殊处理、复杂运算、魔法值进行注释

## 五、分层级注释执行规范（核心）

### 5.1 文件头注释规范

**适用范围**：所有独立可执行的代码文件、模块入口文件

**强制要求**：每个代码文件开头必须添加文件头注释，使用对应语言的块注释格式，禁止使用单行注释拼接

**必填内容项**：

1. 文件核心功能与职责描述（一句话精准概括，复杂文件可补充核心能力清单）
2. 作者信息与创建时间
3. 最后修改时间与修改人（可选，团队协作项目必填）
4. 版权声明/开源协议（开源项目/企业项目必填）
5. 核心依赖与环境约束（可选，特殊依赖场景必填）

**格式示例**：

```python
"""
@File    : user_service.py
@Desc    : 用户核心业务服务层，提供用户注册、登录、信息修改、权限校验全流程能力
@Author  : 开发团队
@Created : 2026-01-01
@Modified: 2026-04-01
@Copyright: Copyright (c) 2026 XX公司. All rights reserved.
@License : MIT
@Dependency: 依赖user_model.py数据模型、auth_utils.py权限工具类
"""
```

### 5.2 模块/包注释规范

**适用范围**：Python包、Go Module、Java Package、C# 命名空间等模块化单元

**强制要求**：模块入口文件（`__init__.py`、`mod.go`等）必须添加模块级注释，说明模块整体能力

**必填内容项**：

1. 模块的核心职责与定位
2. 对外暴露的核心接口/类/函数清单
3. 模块适用场景与使用限制
4. 模块间依赖关系与层级说明

### 5.3 类/结构体注释规范

**适用范围**：所有类、结构体、枚举、接口定义

**强制要求**：每个对外暴露的公共类/结构体必须添加完整块注释，私有内部类必须添加核心功能注释

**必填内容项**：

1. 类/结构体的核心职责与设计意图
2. 适用场景与使用限制
3. 线程安全说明（多线程场景必填）
4. 继承关系与实现接口说明
5. 核心成员变量的含义与约束（复杂类必填）

**格式示例**：

```java
/**
 * 用户信息实体类，封装用户核心基础数据与状态字段
 * 用于用户数据的传输、持久化与业务逻辑处理，仅支持get/set方式操作字段
 * 线程安全：不可变对象，线程安全
 * @author 开发团队
 * @date 2026-04-01
 */
public class UserInfo {
    // 类成员与方法
}
```

### 5.4 函数/方法注释规范

**适用范围**：所有函数、方法、构造函数、析构函数定义

**强制要求**：

- 对外暴露的公共API/接口方法，必须添加完整的文档级块注释，无遗漏必填项
- 内部私有方法，必须添加核心功能注释，复杂方法需补充参数与返回值说明
- 构造函数、重载方法、重写方法必须添加注释，说明与原方法的差异

**必填内容项**：

1. 函数/方法的核心功能，一句话精准概括
2. 入参说明：参数名、数据类型、含义、取值约束、默认值、是否必填
3. 返回值说明：数据类型、含义、异常场景下的返回值
4. 抛出异常/错误说明：异常类型、触发条件、处理建议

**可选补充项**：

- 使用示例与调用方式
- 性能提示与时间复杂度
- 线程安全说明
- 弃用声明：弃用版本、弃用原因、替代方案
- 注意事项与踩坑提示

**格式规范**：严格遵循对应语言的官方文档注释标准，如Python的Google风格、Java的Javadoc、Go的Godoc、JS/TS的JSDoc、Rust的Rustdoc

### 5.5 逻辑块注释规范

**适用范围**：循环、条件分支、异常处理、核心算法块、业务逻辑段

**强制要求**：仅对逻辑复杂、分支场景特殊、非显而易见的代码块添加注释，禁止对简单代码块强行加注释

**注释核心内容**：

1. 代码块的核心处理目标与业务场景
2. 分支判断的设计意图与覆盖场景
3. 循环的终止条件、迭代目的与边界约束
4. 异常处理的覆盖场景与兜底策略
5. 核心算法的原理说明与参考依据

**格式要求**：

- 块注释放在代码块上方，与代码块缩进层级一致
- 使用对应语言的单行注释符，复杂块可使用块注释
- 禁止在代码块中间穿插注释，打断代码逻辑的连贯性

**正反示例**：

```go
// 反面示例：重复代码语义，无价值
if user.Status == 1 {
    // 如果用户状态是1
    return true
}

// 正面示例：解释设计意图，补充业务信息
// 仅允许状态为已激活的用户登录，禁用/未实名用户直接拦截
if user.Status == UserStatusActivated {
    return true
}
```

### 5.6 行内注释规范

**适用范围**：单行代码的特殊说明，禁止滥用

**强制使用场景**（仅以下场景可添加行内注释）：

1. 非显而易见的魔法值、常量定义，补充含义与取值依据
2. 复杂的数学运算、位运算、逻辑运算，补充运算目的
3. 跨版本/跨环境/跨浏览器的兼容处理，补充兼容场景与原因
4. 临时的Workaround、临时解决方案，补充临时原因与清理时间
5. 性能优化的特殊写法，补充优化点与常规写法的差异
6. 容易踩坑的边界处理、空值处理、类型转换，补充风险提示

**格式要求**：

- 行内注释与代码之间至少保留2个空格
- 使用对应语言的单行注释符，禁止使用块注释
- 语言简洁精准，控制在一行内，禁止换行
- 禁止重复代码语义，禁止无意义注释

**正反示例**：

```python
# 反面示例：无效冗余注释
count += 1  # 计数加1

# 正面示例：补充代码无法体现的信息
count += 1  # 统计重试次数，最大重试次数限制为3次，超过触发熔断
```

### 5.7 特殊标签注释规范

**适用范围**：待办事项、问题修复、注意事项、弃用标记

**强制格式**：`标签名: 核心内容 + 责任人 + 截止时间/版本`

**标准标签与使用规则**：

| 标签名 | 适用场景 | 强制要求 |
|--------|----------|----------|
| TODO | 待实现的功能、待优化的逻辑 | 必须补充实现内容、责任人、截止时间 |
| FIXME | 已知的bug、问题代码，需要修复 | 必须补充问题描述、责任人、修复截止时间 |
| NOTE | 重要注意事项、设计说明、踩坑记录 | 必须补充核心提示内容，禁止无意义标注 |
| DEPRECATED | 弃用的代码、方法、类 | 必须补充弃用版本、弃用原因、替代方案 |
| HACK | 非常规的特殊处理、黑科技写法 | 必须补充处理原因、适用场景、风险提示 |

## 六、主流编程语言专属规范

Agent执行时，必须优先遵循对应编程语言的官方注释规范，本Skill为通用补充，不与官方规范冲突：

1. **Python**：严格遵循PEP 8注释规范，文档字符串遵循PEP 257，优先使用Google风格文档字符串，兼容NumPy、reStructuredText风格
2. **Java**：严格遵循Javadoc规范，使用`/** */`块注释，支持IDE自动解析生成API文档
3. **Go**：严格遵循Godoc规范，包注释必须完整，注释直接紧邻定义上方，无空行，使用完整句子，支持官方文档工具解析
4. **JavaScript/TypeScript**：严格遵循JSDoc规范，完整标注类型、参数、返回值、异常，支持TS类型推导与IDE提示
5. **C/C++**：遵循Doxygen规范，公共接口必须完整注释，内部实现注释精准，区分`//`单行注释与`/* */`块注释的使用场景
6. **Rust**：严格遵循Rustdoc规范，使用`///`文档注释，支持Markdown格式，示例代码可自动测试
7. **C#**：严格遵循XML文档注释规范，使用`///`三斜杠注释，完整标注标签，支持IDE智能提示与文档生成

## 七、绝对禁止行为清单

Agent执行本Skill时，绝对禁止出现以下行为，违者视为执行失败：

1. 禁止生成**无效冗余注释**：完全重复代码语义的注释，如`a = 1 # 给a赋值1`
2. 禁止生成**错误/误导性注释**：注释与代码逻辑不符，如代码实现升序，注释标注为降序
3. 禁止**过度注释**：逐行给简单代码添加注释，淹没核心逻辑，增加阅读成本
4. 禁止使用**模糊、歧义、口语化表述**：如`这里处理一下`、`大概是这个意思`、`懂的都懂`
5. 禁止在注释中添加**无关内容**：个人吐槽、无关业务信息、保密信息、网络梗
6. 禁止在注释中**硬编码敏感信息**：密码、密钥、内网地址、个人隐私信息、企业保密数据
7. 禁止使用**未定义的缩写/黑话**：除非是行业通用、团队统一的缩写，且首次出现有完整说明
8. 禁止**修改原有代码逻辑**：仅可操作注释内容，不得修改代码的任何功能、格式、变量与结构
9. 禁止**破坏代码格式与缩进**：注释必须与对应代码的缩进层级一致，不得打乱代码排版
10. 禁止生成**过期注释**：修改代码后必须同步更新注释，不得出现注释与代码版本不一致的情况
11. 禁止使用**拼音、中英文混杂**的注释，除非是行业通用术语，必须使用规范中文/英文
12. 禁止**注释嵌套注释**，避免出现编译错误、语法报错

## 八、注释质量自检标准

Agent完成注释生成后，必须完成以下自检，全部达标方可输出结果：

1. **可读性校验**：新手开发者可通过注释理解代码的整体功能与核心逻辑，无需逐行拆解代码
2. **精准性校验**：注释100%匹配代码逻辑，无错误、无歧义、无误导性内容
3. **规范性校验**：完全符合对应编程语言的官方规范与本Skill的执行要求，格式统一、风格一致
4. **简洁性校验**：无冗余、无重复、无废话，每一条注释都具备明确的信息价值
5. **完整性校验**：文件头、公共类、公共方法的必填注释项无缺失，核心逻辑块注释完整
6. **可维护性校验**：注释可帮助后续开发者快速定位修改点，理解设计意图，降低维护成本
7. **语法校验**：注释无语法错误、无错别字、无标点错误，语句通顺专业
8. **编译校验**：注释不会导致代码出现编译错误、语法报错、IDE警告

## 九、特殊场景处理规则

1. **开源项目代码**：优先遵循项目原有的注释规范与风格，保持与项目整体一致，仅补全缺失的规范注释、修正错误注释，不得强行替换原有注释风格
2. **团队协作项目**：优先遵循团队已有的注释规范文档，本Skill仅作为补充规范，不得与团队规范冲突
3. **教学/示例代码**：可适当放宽注释粒度，增加更多原理说明、步骤拆解、知识点补充，帮助学习者理解，兼顾规范性与教学性
4. **底层库/框架代码**：重点注释对外接口的契约、入参约束、返回值、线程安全、性能边界、使用限制，内部实现注释简洁，聚焦核心算法与设计意图
5. **临时/调试代码**：必须添加注释说明临时用途、过期时间、清理条件，禁止上线代码中存在无标注的临时调试代码
6. **加密/安全相关代码**：仅注释功能与使用方式，禁止注释加密算法的核心逻辑、密钥处理细节、安全绕过方案，避免安全风险
7. **多语言混合代码**：按不同语言的规范分别处理，同一种语言内注释风格保持统一

## 十、执行优先级说明

当存在多套规范时，Agent必须按以下优先级执行，高优先级规范覆盖低优先级规范：

1. 用户明确指定的注释规范/个性化要求
2. 对应编程语言的官方注释规范
3. 目标项目/代码原有的注释风格与规范
4. 团队/企业统一的代码规范文档
5. 本Skill定义的通用注释规范
