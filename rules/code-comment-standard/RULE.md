# 代码注释标准 Rule

**Rule ID**: CODE-COMMENT-RULE-V1.0
**作用时机**: Agent编写代码时同步生成注释

## 核心原则

1. **同步生成**：写代码时同步写注释，注释即代码
2. **风格一致**：既有风格规范时遵循之；不规范时遵循语言官方规范，禁止迎合低质量既有风格
3. **逻辑优先**：明确说明代码执行逻辑与流程，避免完全不描述任何代码逻辑
4. **面向维护者**：重点补充设计意图、踩坑记录、边界约束，同时清晰说明代码执行流程
5. **同步更新**：改代码必须同步改注释

## 分层注释

| 层级 | 场景 | 核心要素 |
|------|------|----------|
| 文件头 | 所有代码文件 | 功能描述、作者、时间、版权 |
| 类/结构体 | 公共类 | 职责、设计意图、线程安全 |
| 函数/方法 | 公共API | 功能一句话、执行流程说明、入参、返回值、异常 |
| 逻辑块 | 分支/循环/算法 | 执行逻辑说明、设计意图、分支覆盖、终止条件 |
| 行内 | 魔法值、复杂运算、兼容处理 | 含义、原因、风险提示 |

## 语言规范

Python → PEP 257 / Google风格 | Java → Javadoc | Go → Godoc | JS/TS → JSDoc | Rust → Rustdoc | C# → XML Doc

## 禁止行为

- ❌ **完全不描述代码逻辑**：避免注释仅解释「为什么」而完全不说明「做了什么」
- ❌ 冗余注释（`a = 1 # 给a赋值1`）
- ❌ 错误/误导性注释
- ❌ 过度注释（逐行加简单注释）
- ❌ 口语化（`这里处理一下`）
- ❌ 敏感信息硬编码
- ❌ 拼音混杂
- ❌ 过期注释

## 特殊标签

`TODO: 内容+责任人+时间` | `FIXME: 问题+修复时间` | `NOTE:` | `DEPRECATED:` | `HACK:`

## 执行优先级

用户指定 → 语言官方规范 → 项目既有风格 → 本规则
