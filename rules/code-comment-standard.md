# 代码注释标准规则

## 核心原则
- 先理解代码，后生成注释，禁止断章取义
- 若代码已有注释风格，优先遵循原有风格
- 仅操作注释内容，禁止修改代码逻辑
- 注释面向未来维护者，补充代码无法体现的信息

## 分层注释要求
- **文件头**: 仅模块入口和独立可执行文件使用，添加前必须询问用户
- **类/结构体**: 公共类必须完整注释，说明职责、场景、线程安全
- **函数/方法**: 公共API必须完整文档注释，说明功能、参数、返回值、异常
- **逻辑块**: 仅复杂、非显而易见的代码块添加注释
- **行内**: 仅魔法值、复杂运算、兼容处理、临时方案使用

## 绝对禁止
- 禁止完全不描述代码逻辑
- 禁止无效冗余注释（重复代码语义）
- 禁止错误/误导性注释
- 禁止过度注释（逐行给简单代码加注释）
- 禁止模糊、歧义、口语化表述
- 禁止修改原有代码逻辑
- 禁止注释与代码版本不一致

## 质量标准
- 新手可通过注释理解整体功能与核心逻辑
- 注释100%匹配代码逻辑，无错误无歧义
- 符合对应编程语言官方规范
- 无冗余、无重复、每一条都有信息价值
